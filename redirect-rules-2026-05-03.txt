DEPLOY STEPS
================================================================================

1. Open AWS Console → CloudFront → Functions.
2. If a viewer-request function already exists on the distribution serving
   discovercloudcroft.com, edit it and merge the redirect map into the existing
   handler (don't overwrite — preserve any www→non-www logic already there).
3. If no function exists yet, create a new one named e.g. "dcc-redirects",
   paste the JavaScript above, save.
4. Publish the function.
5. Attach it to the CloudFront distribution: Behaviors tab → edit the default
   behavior (or each behavior) → Function associations → Viewer Request →
   select the function → save.
6. Wait for distribution status to return to "Deployed" (typically 5–10 min).
7. Test:
     curl -I https://discovercloudcroft.com/stay/stay-the-lodge-at-cloudcroft-2026.html
   Expect: HTTP/2 301 with location header pointing to the new URL.
8. Resubmit the sitemap to Google Search Console after deploy.

NOTES
================================================================================

- All redirects use 301 (permanent) so search engines transfer link equity
  from old URL to new URL.
- 1-year browser cache (max-age=31536000) is safe because these redirects
  are permanent. If you ever rename a page back, bump the cache version or
  invalidate the CloudFront cache.
- Year-specific events (oktoberfest-2026.html, christmas-in-cloudcroft-2026.html,
  etc.) intentionally KEEP the year suffix — they document the 2026 instance
  and will be replaced by -2027 siblings next cycle. No redirect needed.
- The /mexican-canyon-trestle.html root redirect is bundled here because it
  was already on the punch list as a separate item — same deploy, same fix.
