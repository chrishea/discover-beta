WHAT WE TRIED AND WHY IT DIDN'T WORK
================================================================================

ATTEMPT 1 — Custom Domains UI: "Setup redirect" checkbox
  The Amplify Custom Domains panel offers exactly ONE redirect direction
  via a checkbox: "Setup redirect from https://discovercloudcroft.com to
  https://www.discovercloudcroft.com" — i.e., apex → www. There is NO
  matching checkbox for the inverse direction (www → apex).

  We briefly enabled this checkbox to test. Result: apex 302-redirected to
  www, which contradicts every canonical tag on the site. Reverted.

ATTEMPT 2 — Rewrites and redirects panel: host-based JSON rule
  Added rule:
    {
      "source": "https://www.discovercloudcroft.com/<*>",
      "target": "https://discovercloudcroft.com/<*>",
      "status": "301",
      "condition": null
    }

  Amplify ACCEPTED the rule and saved it without errors. The rule appeared
  correctly in the Rewrites and redirects table.

  But the rule SILENTLY DID NOT FIRE. Test against a fresh-path on www:
    curl -sI "https://www.discovercloudcroft.com/totally-random-path-XXX"
    → HTTP/2 301
    → location: /totally-random-path-XXX/        ← Amplify default behavior
    → x-cache: Miss from cloudfront              ← confirmed origin response

  The 301 that fires is Amplify's default `.html`-stripping rule (relative
  path, same hostname). Our host-based redirect was ignored.

  This is consistent with multiple AWS user-forum reports that Amplify
  Hosting Gen 1's redirect engine is path-only — `https://hostname/`
  source patterns are accepted but never matched at request time.

  We removed the dead rule from the panel after confirming this.


WHY OPTION A IS DEFENSIBLE
================================================================================

1. Every page on the site has a canonical tag like:
     <link rel="canonical" href="https://discovercloudcroft.com/PATH">
   This tells Google explicitly that the apex URL is the authoritative
   version. Google's algorithm trusts canonical hints heavily.

2. Sitemap.xml, JSON-LD URLs, og:url, and twitter:url all point to apex.

3. Internal site links use relative paths (no hostname), so the user's
   currently-served hostname stays consistent for that visitor session.

4. The site is brand-new — minimal external backlinks pointing at www to
   "split" link equity. The cost of dual-serving today is near-zero.

5. The fix that would actually work (Option B below) requires touching
   ~96+ files and is reversible. We can defer until/unless GSC reports
   show www indexing causing real problems.


FUTURE OPTIONS (IF SITUATION CHANGES)
================================================================================

OPTION B — Switch the canonical to www (~2-3 hours)
  If Google Search Console eventually reports duplicate-content issues
  or shows www variants ranking instead of apex variants:

  1. Bulk find/replace `https://discovercloudcroft.com/` →
     `https://www.discovercloudcroft.com/` across every:
       - HTML file (canonical, og:url, twitter:url, JSON-LD, hardcoded
         absolute references in body)
       - sitemap.xml
       - llms.txt
       - any redirect rules that reference the apex hostname

  2. Re-enable the "Setup redirect from apex to www" checkbox in the
     Amplify Custom Domains panel (the same one we unchecked).

  3. Verify:
       curl -sI https://discovercloudcroft.com/  → 301 → www variant
       curl -sI https://www.discovercloudcroft.com/  → 200

  This works because Amplify's apex→www redirect IS supported natively,
  and switching the canonical aligns the site with what Amplify can do.

OPTION C — Lambda@Edge override (advanced, fragile)
  Use a CloudFront Function on the Amplify-managed distribution to
  forcibly redirect www→apex before Amplify's logic runs. Bypasses the
  Amplify redirect-engine limitation but introduces:
    - Maintenance burden (Amplify deploys may overwrite the function)
    - Edge debugging complexity
    - Coordination with Amplify support if it conflicts
  Only worth attempting if Option B is somehow unworkable.


HOW TO RECHECK THIS DECISION OVER TIME
================================================================================

Run periodically (quarterly, or after any major SEO change):

  curl -sI https://discovercloudcroft.com/
  curl -sI https://www.discovercloudcroft.com/

If both still return 200 → status quo is intact.

In Google Search Console, watch for:
  - "Duplicate without user-selected canonical" warnings
  - www variants of your URLs showing up in performance reports
  - Coverage drops on apex URLs

If those signals appear, time to execute Option B.
